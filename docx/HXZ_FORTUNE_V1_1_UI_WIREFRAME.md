# HXZ Fortune v1.1 UI 草图说明

最后更新：2026-03-29
对应版本：v1.1
文档用途：用于产品评审、前端实现和联调对齐

---

## 一、设计目标

1. 普通用户 30 秒内理解如何开始输入。
2. 结果表达先结论后细节，降低认知负担。
3. 全流程语气温和、非命令式。
4. 在移动端单手操作场景下可用且不拥挤。

---

## 二、全局 UI 规范（线框层）

### 布局与容器

1. 页面最大内容宽度：980px（桌面），移动端为全宽。
2. 卡片化布局：每个业务块单独卡片，卡片间留有固定纵向间距。
3. 顶部固定轻量导航条（Logo/标题 + 历史入口）。

### 样式语义（不限定具体颜色值）

1. 主色：用于主按钮和关键状态。
2. 辅助色：用于说明与提示。
3. 风险提示色：用于“本段偏离整体趋势”等提醒。
4. 成功反馈色：用于“结果已生成”等状态。

### 组件约束

1. 主要按钮高度 >= 44px。
2. 可点击区域最小 >= 40px。
3. 表单输入采用统一标签、占位、校验提示样式。
4. 所有错误提示出现在字段下方，不用弹窗打断。

## 三、页面结构草图

## 页面 A：输入页（/）

### A1. 信息架构

从上到下：

1. 顶部栏
- 左：产品名称
- 右：历史记录按钮

2. Hero 区（简短引导）
- 标题：一句产品价值说明
- 副文案：强调“可先得到基础结论”

3. 基础信息卡
- 姓名（可选）
- 性别（单选）
- 日历类型（阴历/阳历切换）
- 出生日期（日期选择器）
- 出生时辰（时间选择器，可留空）
- 说明文案：留空也可生成基础结论

4. 区间选择卡
- 默认快捷：未来 30 天（默认选中）
- 自定义：起始日期 + 结束日期
- 规则说明：最多 180 天

5. 精度说明卡
- 大概模式与标准模式差异说明
- 缺失时辰时的结果预期说明

6. 主操作区
- 主按钮：开始分析
- 次操作：重置输入

7. 预留能力区
- 登录同步入口占位（即将上线）

## 页面 C：结果页（/result）

### C1. 信息架构

从上到下：

1. 顶部固定免责声明条
- 文案固定显示，不可关闭

2. 总结结论卡
- 一句话结论
- 本次精度标签（低/中）

3. 八字简版卡
- 年柱、月柱、日柱、时柱（缺时辰则显示“未提供”）

4. 五行占比卡
- 五项条形图或雷达图（线框阶段仅占位）

5. 整段建议卡
- 宜做
- 慎做
- 一句提醒

6. 升级提示卡（条件显示）
- 条件：缺失时辰
- 内容：补全时辰后哪些维度可能变化

7. 分段建议列表
- 每段卡片：起止日期 + 宜做/慎做/提醒 + 四维三档
- 若偏离整体趋势，卡片顶部显示提示条

8. 底部操作区
- 重新分析
- 修改区间再分析

## 四、关键状态设计

### 输入页状态

1. 默认状态：主按钮可点击（必填项齐全时）。
2. 校验失败：字段下方红色提示，主按钮禁用。
3. 区间超限：结束日期下方提示“最多支持 180 天”。

### 结果页状态

1. 标准结果：完整卡片链路。
2. 缺时辰结果：增加升级提示卡。
3. 分段偏离：对应分段卡高亮提示。
4. 空结果异常：显示重试与返回输入页按钮。

### 历史抽屉状态

1. 有记录。
2. 无记录空态。
3. 删除确认轻提示（非阻断）。

---

## 五、响应式断点建议

1. >= 1024px：双列布局（结果页可把八字/五行并排）。
2. 768px - 1023px：单列为主，局部双列。
3. <= 767px：严格单列，底部操作按钮纵向堆叠。

---

## 六、交互细节约束

1. 提交后禁用主按钮，避免重复请求。
2. 日期选择变更时即时校验区间是否合法。
3. 所有“建议”文案使用“可考虑/更适合/建议先”句式。
4. 偏离提示要说明“并非冲突错误，而是阶段变化”。

建议组件拆分：

1. InputFormCard
2. DateRangeCard
3. PrecisionHintCard
4. ResultSummaryCard
5. BaziCard
6. WuxingCard
7. AdviceCard
8. SegmentAdviceCard
9. UpgradeHintCard
10. HistoryDrawer
11. DisclaimerBar

状态字段建议：

1. viewState: input | loading | result | error
2. formState: 输入对象
3. resultState: 接口返回对象
4. historyState: 本地历史数组

---

## 八、评审检查点（UI）

1. 首屏是否在 5 秒内让用户理解“填什么、点哪里”。
2. 结果页是否严格遵守“先结论后细节”。
3. 缺时辰时升级提示是否足够清楚但不打断。
4. 偏离提示是否不会引发“结果冲突=系统错误”的误解。
5. 移动端按钮是否易点、文本是否不过长。
