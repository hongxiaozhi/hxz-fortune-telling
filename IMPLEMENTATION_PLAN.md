# XiaozhiBlog 版本规划

**版本号规则**: `v主版本.功能版本.补丁版本`

- 第三位：发布后的补丁修复（如线上发现 bug → v1.1.1、v1.1.2）
- 第二位：完整功能交付（本文档规划范围，测试通过才打 tag）
- 第一位：大版本里程碑（暂无计划）

**提交与 tag 分离原则**：

- 开发过程中只做 `git commit`，不打 tag
- 功能全部完成 + 本地验收通过 → 打 `git tag v1.X.0`
- 上线后发现小问题修复 → 打 `git tag v1.X.1`，以此类推

## 开发流程

1. 开发过程中只做 `git commit`，不打 tag
2. 某个第二位版本全部任务完成 + 本地验收通过 → `git tag v1.X.0` → 推送部署
3. 上线后发现问题 → 修复 → `git tag v1.X.1`，以此类推
4. 下一个功能版本开发时，重复第 1 步
